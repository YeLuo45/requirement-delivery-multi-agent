# PRD — P-20260623-002: V5 runtime control-plane integration

## Background
V4 delivered the control-plane primitives but left four integration seams: runtime LLM wiring did not consume the budget ledger, PR drafts did not surface git-apply diagnostics, the Web panel component was not routed into the dashboard, and ledger reads were path-coupled.

## Features

### F-1 Runtime budget provider wiring
- `buildDeps({ useLlm: true })` uses `buildAgentProviderWithLedger` when a proposal budget ledger is available.
- Budget exhaustion keeps the pipeline deliverable by degrading to mock provider.
- Existing mock-only and legacy env fallback behavior stays intact.

### F-2 PR draft diagnostics
- `formatPrDraft` includes `git apply --check` status and stderr/stdout in the generated PR body.
- Failed patch checks are visible to operators before they open a PR.

### F-3 Web control-plane route
- Dashboard navigation exposes a `/control-plane` page.
- Page renders the existing `ControlPlanePanel` so the feature is discoverable from the main UI.

### F-4 Ledger storage abstraction
- Control-plane cost reads use a reusable storage-root resolver/helper rather than hardcoded ad-hoc paths.
- Web `/api/control-plane/cost/:proposalId` and future CLI callers can share the same helper.
